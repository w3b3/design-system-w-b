01 // Data Ingestion

Feeding the Pipeline

Threat intelligence begins with continuous ingestion from heterogeneous sources — open-source feeds, commercial subscriptions, honeypots, and internal telemetry. Each source arrives in a different format, at a different cadence, with a different trust score.

The ingestion layer is responsible for collection, transport, and initial triage only. No normalization happens here — fidelity to the original source is preserved so downstream steps can make informed decisions about provenance.

02 // Normalization & Enrichment

From Raw Signal to Structured Intelligence

Raw indicators arrive in dozens of incompatible formats — STIX, JSON, CSV, syslog, vendor-specific schemas. The normalization layer converts everything into a canonical indicator object. Enrichment then appends context that no single source provides alone.

Enrichment happens in parallel pipelines to maintain throughput. Each enrichment service is independently scalable and fails gracefully — a missing geolocation annotation never blocks a critical IOC from reaching detection.

Normalization

Schema Mapping

Incoming records are mapped to an internal canonical schema using per-source adapters. Field names, date formats, and type coercions are handled here. Output is always a typed indicator object with required fields guaranteed.

Normalization

Type Classification

Each indicator is classified by type — IP, domain, URL, file hash, email, CVE, YARA pattern. Type drives which enrichment services are invoked and which detection rules are eligible to match against this record.

Enrichment

Geolocation & ASN

IP indicators are annotated with country, region, city, ASN, and hosting provider data using MaxMind + CAIDA datasets updated weekly. Datacenter vs. residential classification informs risk scoring.

Enrichment

Threat Actor Correlation

Indicators are cross-referenced against a graph of known threat actor TTPs and campaigns using MITRE ATT&CK mappings. A match appends actor attribution, campaign ID, and associated technique IDs to the record.

Enrichment — Critical

Confidence Scoring

A composite confidence score (0–100) is computed from source trust, indicator age, corroboration count across independent sources, and historical false-positive rate. This score gates rule evaluation priority.

Enrichment

Historical Context

Each indicator is checked against the last 90 days of the internal TI store. Recurrence, velocity change, and associated incident history are appended. A new indicator seen three times in 24 hours is automatically escalated.

03 // Detection Rules

Intelligence into Action

Enriched indicators flow into a rule evaluation engine that matches against both static signatures and behavioral logic. Rules are versioned, tested in shadow mode before promotion, and continuously tuned using feedback from analyst triage.

Each rule defines its own confidence threshold, source trust minimum, and indicator type constraints. A rule only fires when the incoming indicator satisfies all conditions — reducing alert fatigue while maintaining high-fidelity detection.

C2 Beacon — Known Malicious IP
Critical

Fires when an internal host initiates outbound connections to an IP indicator with a confidence score above threshold and verified C2 attribution. Triggers an automated host isolation workflow and a P1 alert to the on-call analyst.

MATCH indicator.type = "ip" AND indicator.tags CONTAINS "c2" AND indicator.confidence >= 80 AND indicator.source_trust >= 0.7 ACTION: isolate_host, page_oncall, open_incident(P1)
DNS Lookup — Newly Registered Domain
High

Detects DNS resolution attempts to domains registered within the past 14 days that appear on a TI feed. Newly registered domains are a strong signal for phishing infrastructure and fast-flux C2. The rule suppresses after first alert per host per domain within a 4-hour window.

MATCH indicator.type = "domain" AND indicator.age_days < 14 AND indicator.confidence >= 60 AND NOT suppressed(host, domain, 4h) ACTION: block_dns, alert(HIGH), log_full_pcap
File Hash — Known Malware Variant
Medium

Matches SHA-256 hashes observed on an endpoint against a curated malware hash list sourced from VirusTotal, MalwareBazaar, and internal sandbox detonations. Medium severity reflects that file presence may precede execution — the rule initiates a forensic collection task without blocking the host.

MATCH indicator.type = "file_hash" AND indicator.hash_algo = "sha256" AND indicator.confidence >= 50 AND indicator.vt_detections >= 3 ACTION: collect_forensics, alert(MEDIUM), quarantine_file
